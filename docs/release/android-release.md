# Android Release Workflow

## Release branches

- `release/mobile` is the only branch that can produce Google Play artifacts.
- `android-qa.yml` validates debug/release build smoke and emulator launch behavior.
- `android-release.yml` builds the release AAB and publishes to Google Play.

## Required secrets

- `ANDROID_KEYSTORE_BASE64`
- `ANDROID_STORE_PASSWORD`
- `ANDROID_KEY_ALIAS`
- `ANDROID_KEY_PASSWORD`
- `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON`

## Google Play workflow

1. Merge a pull request into `release/mobile`.
2. `android-release.yml` builds the release bundle and publishes to the internal track when Play credentials are present.
3. Promote the same workflow manually with `track=closed` or `track=production` once internal validation is complete.
4. Protect `android-internal` and `android-production` environments with required reviewers before using them for releases.

## Initial Play Console setup

1. Create the Play Console app manually once.
2. Upload the first signed AAB manually if the API reports `Package not found`.
3. Grant the service account app-level release permissions in Play Console.
4. Populate store listing assets, privacy policy, testers, and content rating before production promotion.

## Versioning

- Shared release metadata is generated by `scripts/release/compute-release-metadata.mjs`.
- Android version name and version code are injected into Gradle through environment variables.
- The workflow stores release notes and the AAB as artifacts for auditability.
